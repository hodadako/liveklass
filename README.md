## 1. 프로젝트 개요

- 문제 정의
  - 크리에이터(강사)는 강의를 개설하고 수강 정원, 가격, 기간을 설정합니다.
  - 클래스메이트(수강생)는 원하는 강의에 수강 신청을 합니다.
  - 정원이 초과되면 신청이 불가합니다.
  - 신청 후 결제가 완료되어야 수강 확정됩니다.
  - 수강 확정 후 일정 기간 내에는 취소가 가능하며, 이후에는 불가합니다.

- 과제 구현 범위
  - 강의(Class) 생성, 상태 변경(DRAFT → OPEN → CLOSED), 목록 및 상세 조회
  - 수강 신청(Enrollment) 생성, 결제 확정(PENDING → CONFIRMED), 취소(CANCELLED)
  - 강의 정원 초과 방지 및 동시성 처리
  - 수강 취소 가능 기간 제한
  - 대기열(Waiting Queue) 기반 수강 신청 제어

## 4. 요구사항 해석 및 가정
### 요구사항을 어떻게 해석했는지
  - 요구사항의 배경 시나리오를 기반으로 강의(Class), 수강 신청(Enrollment), 대기열(EnrollmentTicket)과 같은 주요 도메인을 먼저 정의했습니다.
  - 각 도메인이 가질 수 있는 상태와 상태 전이 흐름을 먼저 설계한 후, 이를 검증하는 단위 테스트를 작성했습니다.
  - 이후 도메인 로직을 조합하여 사용자 관점의 기능(수강 신청, 결제 확정, 취소 등)을 구성했습니다.
  - 사용자 요청 흐름보다는 비즈니스 규칙(정원 제한, 상태 전이, 만료 등)을 우선적으로 설계하고, 이를 기반으로 API를 구성했습니다.
### 명확하지 않은 부분에 대한 가정
#### 동시성 제어
- 동시에 2명이 마지막 자리를 신청하면 비관적 락을 통해서 강의와 수강신청에 대한 신청을 막고 한명만 성공할 수 있도록 했습니다.
- 강의와 수강신청이 수정이 자주 일어나지 않고, 그만큼 충돌이 없을거라고 판단하여 비관적 락을 선택 했습니다 (select for update)
- 강의에 대한 수강 신청이 직접 일어나는 부분은 결제 이후 수강 신청이 `CONFIRMED`상태로 변경되면 처리할 수 있도록 바꾸어서, 실제 결제가 일어나는 상황을 가정했습니다.
- 한 사람이 여러번 신청하는 경우를 수강 신청 도메인에서 Unique (강의, 회원) Index를 걸어서 방지하도록 만들었습니다.
#### 결제
- 결제는 별도의 Payment API를 통해 처리하며, 외부 결제 시스템 대신 `PaymentClient`가 단순 지연만 시뮬레이션한 뒤 수강 신청 상태를 `PENDING → CONFIRMED`로 변경하도록 구성했습니다.
- 실패 케이스와 재시도 정책은 현재 고려하지 않았고, 실제 PG 연동 대신 결제 성공 흐름만 단순화해서 구현했습니다.
#### 상태 전이
- 상태 전이는 전부 application layer에서 일관적으로 처리하도록 했습니다.
- 상태변경 시 일어나는 예외는 도메인에 책임을 두어, application layer가 해당 예외를 직접 발행하지 않고 도메인에서 예외가 일어나도록 처리했습니다.
- 상태 전이 규칙은 현재 도메인 내부 if문 분기를 통해 처리됩니다.
#### 취소 정책
- 현재 취소 정책은 처음 수강 신청이 만들어진 시간으로 부터 7일 뒤까지로 설정했습니다.
- 사용자가 취소시 수강 신청의 상태에 따라서 처리를 다르게 했습니다.
  - `PENDING` 일 경우 자리가 늘어나지 않습니다.
  - `CONFIRMED`일 경우 기존에 차지 하던 자리를 반환합니다.
  - 이미 시작된 강의도 취소가 가능하도록 열어두었습니다.
#### 대기열
- 대기열(waitlist)에 대한 요구사항이 명확하게 정의되어 있지 않아 두 가지 해석이 가능하다고 판단했습니다.
  1. 정원 초과 시 수강 신청을 위한 대기 명단
  2. 동시 요청 제어를 위한 트래픽 대기열

- 본 구현에서는 수강 신청 요청을 순차적으로 처리하는 트래픽 제어형 대기열로 해석하여 구현했습니다.
- 이는 동시에 많은 사용자가 마지막 자리를 신청하는 경우 발생할 수 있는 경쟁 상태를 제어하기 위해 구현했습니다.
- 다만 티켓 기반 구조를 사용하였기 때문에 향후 정원 기반 대기열로도 전환이 가능한 형태로 구현했습니다.
- 현재 대기 순번과 대기 인원은 강의별이 아니라 전체 `WAITING` 티켓 기준으로 계산됩니다.
- 동일한 `(classId, memberId)` 조합으로 다시 진입할 경우 새 티켓을 발급하기보다 기존 티켓을 재사용합니다.
- `ALLOWED` 티켓은 허용 시점에 만료 시간이 설정되며, 만료 시간을 지난 뒤 사용을 시도할 때 `EXPIRED` 상태로 전이됩니다. 이후 `EXPIRED` 티켓은 스케줄러가 정리하고 expired SSE 이벤트를 전송합니다.

## 5. 도메인 설계
이 프로젝트는 강의 개설부터 대기열 진입, 수강 신청 생성, 수강 확정, 취소까지의 흐름을 `Class`, `Enrollment`, `EnrollmentTicket`, `Member` 네 가지 핵심 도메인으로 나누어 설계했습니다. 서비스 계층은 요청 흐름을 조합하고, 실제 상태 전이와 규칙 검증은 각 도메인 객체가 직접 책임지도록 구성했습니다.

`Class`는 강의의 기본 정보와 상태, 정원, 현재 확정 인원을 관리합니다. 상태는 `DRAFT → OPEN → CLOSED` 순서로 전이되며, `OPEN` 상태일 때만 수강 신청을 받을 수 있습니다. 또한 수강 확정 시점에 정원 초과 여부를 검증하고, 확정 취소가 일어날 경우 반환할 좌석 수도 함께 관리합니다.

`Enrollment`는 특정 수강생이 특정 강의에 신청한 이력을 의미합니다. 상태는 `PENDING → CONFIRMED → CANCELLED` 흐름을 가지며, 신청 생성 시점에는 우선 `PENDING`으로 저장되고 이후 확정되면 `CONFIRMED`로 변경됩니다. 동일한 사용자가 같은 강의에 여러 번 신청하지 못하도록 `(class, member)` 단위 유니크 제약을 두었습니다.

`EnrollmentTicket`은 대기열 통과를 위해 발급되는 티켓입니다. 상태는 `WAITING → ALLOWED → USED` 또는 `EXPIRED`로 관리되며, 허용된 티켓만 실제 수강 신청에 사용할 수 있습니다. 이 도메인은 정원 기반 waitlist라기보다는, 동시에 몰리는 요청을 순차적으로 흘려보내기 위한 트래픽 제어 역할에 가깝습니다.

`Member`는 시스템 사용자를 표현하며, 역할은 강의를 개설하는 `CREATOR`와 수강 신청을 수행하는 `STUDENT`로 구분됩니다.

## 6. 데이터 모델
![ERD](/docs/erd.png)
현재 구현 기준 데이터 모델은 `member`, `class`, `enrollment`, `enrollment_ticket` 네 개의 핵심 테이블로 구성됩니다. 모든 테이블은 공통 상위 엔티티를 통해 `id`, `created_date`, `updated_date`를 가지며, 생성/수정 시각은 JPA Auditing으로 자동 관리됩니다.

`member` 테이블은 회원 이름과 역할을 저장합니다. 역할 값은 `CREATOR`, `STUDENT` 문자열 enum으로 저장됩니다.

`class` 테이블은 강의 제목, 설명, 상태, 가격, 정원, 현재 확정 인원, 시작/종료 일시를 저장합니다. 강의 상태는 `DRAFT`, `OPEN`, `CLOSED` 중 하나입니다.

`enrollment` 테이블은 수강 신청 정보를 저장하며, `class_id`와 `member_id`를 통해 어떤 강의에 어떤 회원이 신청했는지를 표현합니다. 또한 신청 시각, 확정 시각, 취소 시각을 별도로 보관해 상태 변경 흐름을 추적할 수 있습니다. 동일 사용자의 동일 강의 중복 신청을 방지하기 위해 `(class_id, member_id)` 유니크 제약을 둡니다.

`enrollment_ticket` 테이블은 대기열 티켓 정보를 저장합니다. 티켓 상태와 허용 시각, 만료 시각, 사용 시각을 관리하며, 구현상 `Class`, `Member`와 직접 연관관계를 맺기보다는 `class_id`, `member_id` 값을 보관하는 독립 테이블 형태로 두었습니다. 이 역시 동일 사용자에 대한 중복 티켓 생성을 막기 위해 `(class_id, member_id)` 유니크 제약을 가집니다.

ERD 관점에서 핵심 관계는 `Class 1 : N Enrollment`, `Member 1 : N Enrollment`입니다. `EnrollmentTicket`은 신청 전 단계의 흐름 제어용 테이블로 이해하면 현재 구현과 가장 잘 맞습니다.

## 7. API 목록 및 예시
현재 구현된 API는 강의 관리, 대기열 진입, 수강 신청 생성, 별도 결제 확정, 내 신청 목록 조회 및 취소를 중심으로 구성되어 있습니다.

### 강의 API
- `GET /api/v1/classes`
  - 강의 목록을 조회합니다.
  - `status` 쿼리 파라미터를 사용하면 상태별 필터링이 가능합니다.
- `GET /api/v1/classes/{classId}`
  - 특정 강의의 상세 정보를 조회합니다.
- `POST /api/v1/classes`
  - 새로운 강의를 생성합니다.
- `PATCH /api/v1/classes/{classId}/status`
  - 강의 상태를 `OPEN` 또는 `CLOSED`로 변경합니다.

예시)
```http
POST /api/v1/classes
Content-Type: application/json

{
  "title": "Spring Boot 입문",
  "description": "기초부터 배우는 스프링 부트",
  "price": 100000,
  "capacity": 30,
  "startDate": "2026-05-01T10:00:00",
  "endDate": "2026-05-31T18:00:00"
}
```

### 대기열 API
- `POST /api/v1/classes/{classId}/waiting-tickets`
  - 특정 강의에 대한 대기열 티켓을 발급받습니다.
- `GET /api/v1/waiting-tickets/{ticketId}`
  - 현재 티켓의 전체 WAITING 큐 기준 대기 순번, 대기 인원, 예상 대기 시간을 조회합니다.
- `GET /api/v1/waiting-tickets/{ticketId}/events`
  - SSE로 티켓 허용 이벤트를 구독하고, 이미 `EXPIRED` 상태가 된 티켓 정리 시 expired 이벤트를 받을 수 있습니다.

예시)
```http
POST /api/v1/classes/1/waiting-tickets
Content-Type: application/json

{
  "memberId": 10
}
```

### 수강 신청 API
- `POST /api/v1/classes/{classId}/enrollments`
  - 허용된 대기열 티켓을 사용해 수강 신청을 생성합니다.
- `GET /api/v1/classes/{classId}/enrollments`
  - 특정 강의의 수강 신청 목록을 조회합니다.
- `GET /api/v1/enrollments?memberId={memberId}`
  - 특정 회원의 내 수강 신청 목록을 조회합니다.
- `DELETE /api/v1/enrollments/{enrollmentId}?memberId={memberId}`
  - 본인의 수강 신청을 취소합니다.

예시)
```http
POST /api/v1/classes/1/enrollments
Content-Type: application/json

{
  "memberId": 10,
  "ticketId": 15
}
```

### 결제 API
- `POST /api/v1/payments/enrollments/{enrollmentId}/confirm`
  - 결제를 성공 처리하고 해당 수강 신청을 `CONFIRMED` 상태로 변경한 결과를 응답으로 반환합니다.

예시)
```http
POST /api/v1/payments/enrollments/10/confirm
```

응답 예시)
```json
{
  "enrollmentId": 10,
  "enrollmentStatus": "CONFIRMED",
  "confirmedDate": "2026-04-02T10:00:00"
}
```

## 8. 비즈니스 규칙 정리
이 구현에서 가장 중요한 규칙은 정원 관리 시점을 수강 신청 생성이 아니라 수강 확정 시점에 두었다는 점입니다. 사용자는 먼저 대기열을 통과한 뒤 `PENDING` 상태의 수강 신청을 만들고, 이후 확정 단계에서만 실제 정원이 차감됩니다. 따라서 신청 자체는 생성되었더라도 확정에 실패하면 좌석은 점유되지 않습니다.

강의는 반드시 `OPEN` 상태여야 신청할 수 있으며, 신청 시에는 해당 강의와 회원에 대해 발급된 `ALLOWED` 상태 티켓이 필요합니다. 허용된 티켓은 5분 동안 유효하며, 만료 시간을 지난 뒤 사용을 시도하면 `EXPIRED` 상태로 전이되어 더 이상 신청에 사용할 수 없습니다.

동일 사용자는 같은 강의에 중복 신청할 수 없으며, 대기열 재진입 시에는 새 티켓 대신 기존 티켓을 재사용합니다. 또한 수강 취소는 신청자 본인만 가능하며, 취소 가능 기간은 `requestedDate` 기준 7일 이내로 제한했습니다. 취소 시점의 상태에 따라 처리도 달라집니다. `PENDING` 상태를 취소하면 정원 변화는 없고, `CONFIRMED` 상태를 취소하면 기존에 차지하고 있던 좌석이 반환됩니다.

동시성 제어는 비관적 락을 사용해 처리했습니다. 마지막 한 자리를 두고 여러 요청이 동시에 들어오는 경우, 동일 티켓이 중복 사용되는 경우, 이미 취소된 신청을 다시 취소하는 경우에도 정합성이 깨지지 않도록 관련 레코드를 잠근 뒤 순차적으로 처리하도록 했습니다.

### 대기열 흐름 정리
1. 사용자는 수강 신청을 바로 생성하지 않고, 먼저 특정 강의에 대한 대기열 티켓을 발급받습니다. 이때 티켓은 `WAITING` 상태로 생성되며, 아직 실제 수강 신청을 수행할 수는 없습니다.

2. 이후 스케줄러가 주기적으로 `WAITING` 상태의 티켓을 조회하고, 처리 가능한 티켓을 `ALLOWED` 상태로 변경합니다. 티켓이 허용되면 서버는 SSE를 통해 사용자에게 신청 가능 이벤트를 전송합니다.

3. 사용자는 `ALLOWED` 상태의 티켓을 가지고 수강 신청 API를 호출합니다. 이때 티켓이 해당 강의와 회원에게 발급된 것인지, 아직 만료되지 않았는지, 이미 사용되지 않았는지를 검증합니다.

4. 검증이 끝나면 티켓은 `USED` 상태로 변경되고, 수강 신청은 `PENDING` 상태로 생성됩니다. 이 단계에서는 아직 정원이 차감되지 않으며, 실제 좌석 점유는 결제 확정 단계에서만 발생합니다.

5. 이후 사용자가 결제를 확정하면 수강 신청 상태가 `PENDING`에서 `CONFIRMED`로 변경되고, 이 시점에 강의의 확정 인원이 증가합니다. 반대로 결제가 확정되지 않거나 신청이 취소되면 좌석은 점유되지 않습니다.

6. 허용된 티켓은 5분 동안만 사용할 수 있습니다. 만료 시간이 지난 티켓으로 수강 신청을 시도하면 해당 티켓은 `EXPIRED` 상태로 변경되며, 더 이상 신청에 사용할 수 없습니다. 또한 스케줄러는 만료된 티켓을 주기적으로 정리하고, 필요한 경우 SSE를 통해 만료 이벤트를 전송합니다.

7. 따라서 전체 흐름은 `WAITING → ALLOWED → USED` 또는 `EXPIRED`로 정리할 수 있으며, 수강 신청은 반드시 `ALLOWED` 상태의 티켓을 통해서만 생성됩니다.


## 9. 설계 결정과 이유
애플리케이션 구조는 `presentation`, `application`, `domain`, `persistence` 레이어로 분리했습니다. 컨트롤러는 HTTP 요청과 응답만 다루고, 서비스는 유스케이스를 조합하며, 실제 상태 전이 규칙과 핵심 비즈니스 검증은 도메인 객체 내부에 두었습니다. 이렇게 분리하면 요청 방식이 바뀌더라도 핵심 규칙은 도메인 계층에서 일관되게 유지할 수 있습니다.

수강 신청과 수강 확정을 분리한 이유는, 실제 서비스에서는 신청 이후 결제나 추가 검증 같은 후속 절차가 개입될 수 있기 때문입니다. 먼저 `PENDING` 신청을 만들고 이후 별도의 Payment API를 통해 결제 성공을 처리한 시점에만 좌석을 점유하게 하면, 정원 관리 시점과 결제/확정 시점을 명확히 구분할 수 있습니다.

동시성 제어 방식으로는 낙관적 락보다 비관적 락을 선택했습니다. 이 프로젝트의 핵심 경쟁 구간은 마지막 좌석 확정, 동일 신청 중복 처리, 동일 티켓 재사용 방지처럼 충돌이 발생하면 즉시 차단해야 하는 상황이기 때문에, 읽는 시점부터 레코드를 잠그는 방식이 더 직접적이라고 판단했습니다.

  - 수강 신청 → 확정 → 취소 흐름 검증

  - 마지막 좌석에 대해 여러 요청 동시 처리
  - 하나만 성공하는지 검증

## 11. 미구현 / 제약사항
### 미구현
- 어플리케이션 전반적인 로깅 및 모니터링
- 사용자 입력값에 대한 상세 유효성 검증 (Bean Validation 등)
- 외부 결제 시스템 연동
- 인증/인가 처리 (JWT, 세션 등)
### 제약사항
#### 대기열
- 사용자는 대기열에 진입하면 대기 티켓을 발급받고, SSE를 통해 티켓 상태 변경 이벤트를 받을 수 있습니다.
- 다만 SSE는 네트워크 끊김이나 재연결 실패로 이벤트 유실 가능성이 있기 때문에, 현재 티켓 상태는 조회 API를 통해서도 확인할 수 있도록 구현했습니다.
- 현재 대기열은 전체 강의와 전체 회원을 대상으로 하나의 스케줄러가 대기 티켓을 조회하고 상태를 전이하는 방식으로 구현되어 있습니다.
- 이 구조는 구현이 단순하지만, 대기 티켓 수가 많아질 경우 스케줄러가 병목이 될 수 있습니다.
- 병목이 발생할 경우 강의 단위로 워커를 분리하거나, 여러 워커가 처리할 수 있도록 구조를 확장할 수 있습니다.
- 다만 워커를 여러 개로 늘릴 경우 동일 티켓이 중복 처리되지 않도록 락, 상태 조건 업데이트와 같은 동시성 제어가 필요합니다.
- 현재 대기열은 DB 기반 큐로 구성되어 있어 주기적으로 DB를 조회해야 합니다.
- 서버를 여러 대로 스케일 아웃할 경우 각 서버의 스케줄러가 같은 티켓을 동시에 처리할 수 있으므로, 분산 환경을 고려한 락 또는 작업 선점 방식이 필요합니다.
- 트래픽이 더 커지는 경우 Redis, 메시지 큐, Kafka와 같은 별도 큐 시스템으로 확장하는 방안을 고려할 수 있습니다.
